# Client-clean / scrub (Stage 11) — F-013 · F-014 · F-015

Read when running Stage 11, the **last** pipeline stage and a **BLOCKING** gate: the build is not "done"
until the **client** repo is clean of all build apparatus. The deliverable is the brand's own canon — it must
read as if the brand authored it, carrying no trace of the tool, the method, or any reference brand.

> Scope note: this scrubs the **client** repo the builder produces. It never touches this builder repo's own
> docs (we keep our attribution); the source templates are kept brand- and tool-neutral so they don't *seed*
> bleed in the first place.

## 1. Tool self-attribution (F-015)

The client `README.md` / `RESIDENT.md` / `CLAUDE.md` and any other doc must **not** credit the build tool or
skill. Strip every:

- "Built with / scaffolded with / generated by `brand-system-skills`" line,
- link back to the tool/skill repo,
- "brand-canon-builder" / "brand-canon-scoper" / skill credit or generator banner.

Grep the client tree for `brand-system-skills`, the skill names, and the tool repo URL — there must be **zero
hits** before "done". (The source templates are kept neutral so this is a runtime backstop, not a routine
fix.)

## 2. Reference-brand bleed (F-013) — a RUNTIME catch

The templates are brand-blind, so bleed is never *in* the templates — it is whatever the **build imported**
from a method / reference / example brand instead of the client's own material. Audit the generated client
canon for any **rule, name, example, or value not traceable to the client's own material**:

- naming rules and glossary entries carried over from a reference canon,
- example names / sample copy / placeholder entities from the method or an example brand,
- color / type / spacing values that trace to a reference brand rather than the client's extracted assets.

**The test:** every line in the client canon must trace to the client's material (or be an explicit, honest
GAP). If it traces to a method/reference/example brand instead, **strip it**. A rule that only ever made
sense for the reference brand does not belong in this client's canon.

## 3. {{PLACEHOLDER}} leftovers + scaffolding chatter

The client deliverable must read as finished:

- **No `{{...}}` placeholders survive.** Grep the client tree for `{{` — every one is either filled from the
  client's material or converted to an honest GAP; none ship raw.
- **Remove GUIDE / instructional comments** — e.g. `<!-- GUIDE: … -->`, `<!-- BUILDER: … -->` and any
  template chatter that addresses the builder rather than the brand owner. They are scaffolding, not content.
- Remove dead template sections that were never filled and are not tracked as a GAP.

## 4. Ratification (F-014) — never auto-stamp

Ratification is a record of a **real human decision**, never a side effect of a build:

- **Never auto-write** "Ratified by…", "Approved by…", or flip a gap to `CLOSED (ratified)`.
- The default state of every owner decision is **unratified-pending**.
- A gap moves to `CLOSED (ratified)` and the canon stamps a ratifier **only on real owner sign-off** — which
  is the human PR review assembled at Stage 10 (the human gate). This stage enforces that gate: it does not
  manufacture the sign-off it was waiting for.

## 5. The BLOCKING gate

The build reaches "done" only when the client repo is clean:

- zero tool/skill attribution (§1),
- zero reference-brand bleed — everything traces to the client's material or an honest GAP (§2),
- zero surviving `{{…}}` and zero GUIDE/builder chatter (§3),
- no auto-stamped ratification; state is unratified-pending until the owner signs off (§4).

Any residue → not done. Clean, then hand off the PR for the owner's human sign-off (Stage 10 gate).
